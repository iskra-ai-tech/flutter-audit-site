02b Read on the page · ten fronts

What the audit actually opens.

Ten engine-level fronts I open against every Flutter codebase. Your symptom binds to one of these — usually two. Each front lists the diagnostic that proves the cause and the test that holds the fix in place after I leave. Public on purpose; if your team is already running every probe below cleanly, you do not need me.

  1. 01

    Cold start to first interactive frame — not first paint.

    “Install funnel drops 12% between download and first session. Competitors open instantly. Ours sits, flashes white, content fades in.”

    TTI is the budget that matters, not first paint. The kill is almost always upstream of runApp — synchronous Firebase init, plugin chains hanging off Application.onCreate, eager SharedPreferences, lazy AOT snapshot mmap stalls. Diagnosed with adb shell am start -W -S -R 5 + a Perfetto trace bracketing FlutterEngine ctor and Dart_LoadScriptFromKernel. FlutterEngineGroup pre-warm on a worker thread reliably reclaims 200–400 ms on mid-tier Android. Bound by an integration_test that fails CI if timeToFirstFrameRasterizedMicros > 800,000 on a Firebase Test Lab Pixel 7 — the test stays in the suite as a permanent regression bound.

  2. 02

    Splash → content visual continuity.

    “There’s a beat where the icon sits, then the screen flashes white, then content fades in. Feels broken.”

    Three frames are involved: OS splash (Android 12+ windowSplashScreenAnimatedIcon, iOS LaunchScreen.storyboard), Flutter’s first Dart frame (usually a blank Scaffold at colorScheme.surface), then real content as routes hydrate. The flash is the frame 0 → 1 discontinuity. Audited by capturing the first ten frames into a per-frame golden set on a pinned device. Fix: pixel-match the native splash drawable to the first Scaffold background, then gate deferFirstFrame() / allowFirstFrame() around precacheImage of the route’s load-bearing assets. Bound by a triple-frame golden test on Pixel 8a + iPhone 13 that fails any PR shifting the cold-launch optics.

  3. 03

    Image pipeline GPU texture residency.

    “App crashes after scrolling a 200-item gallery twice. iOS jetsam, Android low-memory-killer. We’re using cached_network_image, so we thought we were fine.”

    ImageCache has two budgets — count (default 1000) and bytes (default 100 MB). But a decoded ui.Image is a GPU texture handle, and Impeller’s texture atlas runs its own LRU independent of imageCache.evict(). Without cacheWidth/cacheHeight matched to logical pixels × devicePixelRatio, a 4000×3000 JPEG decodes to a 48 MB RGBA8 texture — ten of those is half a gig of GPU memory the Dart side cannot evict. Diagnosed with adb shell dumpsys meminfo (Graphics line) + DevTools Memory live ui.Image count + didHaveMemoryPressure callback wiring. Bound by an integration_test scrolling the gallery 5× with a monotonic-growth assertion on the Graphics line.

  4. 04

    Frame budget split: UI thread vs raster thread.

    “DevTools claims 120 fps but the iPhone still feels stuttery. We can’t see what’s wrong.”

    At 120 Hz the 8.33 ms budget is two parallel budgets, not a sum. SchedulerBinding.addTimingsCallback surfaces buildDuration and rasterDuration as independent phases. A SaveLayer-heavy subtree (Opacity, BackdropFilter, ShaderMask, scaled Transform) inflates raster while UI looks pristine — each SaveLayer becomes an EntityPass with its own render target, and one nested BackdropFilter inside another can produce quadratic intermediate-texture cost. Ship p99 raster as a separate SLI to your APM via FrameTiming sampling. Bound by an integration_test driving a scripted scroll under binding.traceAction asserting 90th_percentile_frame_rasterizer_time_millis < 8.0 and missed_frame_rasterizer_budget_count == 0.

  5. 05

    App size as install-conversion lever.

    “Install conversion in India is half what we see in the US. Users see ‘47 MB’ on the Play Store and bounce. Updates have a 40% lower opt-in rate than peers.”

    Mid-tier Android math: every +1 MB over 30 MB drops install conversion roughly one percentage point. The 2026 audit signal hiding in plain sight: Material Symbols variable font silently defeats --tree-shake-icons because tree-shake walks const IconData call sites and the variable axes defeat per-glyph elimination — teams migrating from Material Icons to Symbols ship 2 MB of unreferenced glyphs and never notice. Audited via flutter build appbundle --analyze-size + bundletool get-size total. Recovery in order: --split-debug-info --obfuscate (~30% off libapp.so), R8 fullMode reinstated, density variant pruning, subsetted icon font, deferred components for once-per-lifetime flows. Bound by a CI size-budget gate that fails any PR exceeding the budget with an attributable treemap diff posted to the PR.

  6. 06

    Battery and thermal pressure on real devices.

    “Users complain the app heats the phone. App Store reviews mention battery drain. We have no idea what’s burning.”

    The render loop is itself an energy line. Over-invalidation on a 120 Hz idle screen sits at ~8% CPU and noticeable mW; a parked Ticker is ~0.5%. Location accuracy tiers fan out by an order of magnitude — high (~80 mW continuous) vs passive (~2 mW). Background work coalescing through WorkManager / BGTaskScheduler, radio-tail cost on network polling, BLE/sensor wake patterns — each one is a measurable line. Thermal headroom (getThermalHeadroom(), iOS ProcessInfo.thermalState) reveals when the SoC starts throttling and Flutter’s raster degrades. Diagnosed with Android Energy Profiler + Battery Historian; CI-bound via a Monsoon rig or instrumented Firebase Test Lab spec asserting per-route mW budgets.

  7. 07

    Memory leaks and soak degradation.

    “Long sessions feel heavier. After 30 minutes of normal use the frame rate drifts down; iOS eventually jetsams.”

    package:leak_tracker (now integrated into flutter_test) detects three leak classes: not-disposed, not-GCed-after-disposed, GCed-late. Canonical 2026 offenders unchanged in shape: AnimationController without dispose, StreamSubscription from Stream.listen() never cancelled, TextEditingController/ScrollController/FocusNode silent leaks, GestureRecognizer constructed in build(). Heap snapshot diff between two scenario boundaries names the retained class; DevTools Memory → Leaks tab emits the retention-path JSON. Bound by every widget test enabling LeakTesting.settings.withTracked(allNotDisposed: true) + a 5-minute soak integration_test asserting heap-growth slope ~ 0 across a scripted user journey.

  8. 08

    Concurrency correctness in Dart 3.

    “Random null deref under heavy network. We patched it. It came back two releases later with a different stack.”

    compute() is no longer the right primitive — Isolate.run with Isolate Groups and TransferableTypedData is the Dart 3 path, with shared-heap semantics that change what’s safe to pass. Real failure modes: microtask starvation of the gesture arbiter under a runaway await chain inside a Stream listener, double-fetch races on rebuild without CancelableOperation, Stream backpressure under asyncMap vs switchMap ordering. Diagnosed with dart:developer Timeline slices + DevTools async-aware stack traces + VM Service getAllocationProfile. Bound by deterministic fake_async scheduler tests + property-based glados tests generating adversarial input sequences against the concurrency invariant.

  9. 09

    Technical debt + legacy code archaeology.

    “Adding one feature touches eight files. Velocity is half what it was a year ago. We have three state-management libraries because three engineers each had a favourite.”

    Five quantifiable decay signals: state-management library count > 1, navigation library count > 1, theme literals outside the token system, custom widget files > 200 LOC, dependency versions frozen > 18 months. Churn × complexity hotspots (Tornhill applied to a Flutter tree) name the five files responsible for 60% of regression incidents. Legacy migration risk surfaces as null-safety partial migrations still in flight, v1 Android embedding holdouts, Material 2 → 3 incomplete sweeps, Impeller opt-out flags still set, abandoned isolates running dead code paths. Tooling: dart analyze --fatal-infos, dcm, pana, osv-scanner, custom AST visitors for project-specific anti-patterns. Bound by mutation score on the domain core (target > 70% via mutation_test) and a regression-bounds/ directory of permanent test guards each pinned to a closed ticket.

  10. 10

    Bug detection where the test is the proof.

    “We close bugs and they come back. QA finds it, the engineer says ‘fixed,’ it ships, two releases later the same crash with a different stack. We have 78% line coverage and zero regression confidence.”

    Coverage is not proof — a thousand expect(true, true) tests show 100% coverage and 0% proof. Audit methodology: every reported issue produces a failing reproducer first, committed before the fix. The fix lands in the same PR that flips the test red → green. The test stays in the suite as a permanent regression bound. Diagnostic substrate: structured NDJSON logs with correlation IDs surviving across isolate boundaries, replayable from a captured production trace. Test taxonomy mapped to proof type: unit (pure logic), widget (with rebuild-count assertion), integration_test (with FrameTiming assertion), golden (platform-pinned), property-based (glados), mutation (mutation_test), soak (heap-slope), telemetry-replay. The audit deliverable is not a PDF of opinions — it is a series of red-to-green commits, each one a permanent guard, replayable by anyone with a checkout.

If two or more of these probes fire dirty against your codebase, the audit usually pays for itself in the first PR your team doesn’t have to write twice.

03 How I read code

A code audit is not a code review.

A code review tells you what is syntactically wrong. An audit tells you what is structurally wrong. Which decision from 2023 is costing a week per release. Which SaveLayer ancestor is taxing every animated frame. Which third of your providers can be deleted without anyone noticing. Which migration started in 2024 and never finished.

A linter sees tokens; the audit sees decisions. I trace narrow error-prone paths through structured logs and engine timelines until the symptom binds to a named cause — over-invalidation, image-pipeline residency, Impeller PSO miss, isolate starvation, ghost rebuild. Every claim in the deliverable lands as a failing test first, then a green one. Diagnosis is the hard part. The fix list is short on purpose. Most of the value is what is no longer there.

06 Questions

Six questions, six answers.

Why an audit instead of hiring a senior Flutter engineer?

Hiring takes three to four months and a salary. The audit takes three days and $500. The audit answers the harder question hiring does not — which senior, with which depth, against which file. If you still need to hire afterwards, the audit becomes the job description.

Do you need the source code?

Yes. The audit is me reading your code. Walkthroughs, screen-shares, and recordings are not substitutes — the diagnosis lives in the repo. NDA before code changes hands. If the source cannot be shared, this is not the engagement to book.

Why $500 — is this a real audit at that price?

Yes. You are paying for diagnostic compression: three days of a senior engineer reading the codebase end to end, plus a ranked fix list, a real-device frame timeline, a stakeholder brief, and a walkthrough with your team. Senior contractor day rates run $1,200–$2,400. The audit prices under day-rate × 3 because the deliverable is bounded — diagnosis, not implementation. Same shape as a $5,000 audit, tighter scope. If the work runs over my budget, that is not your problem.

Our app is not slow — it is just messy. Is the audit still useful?

Often more so. Performance audits get the headlines; architectural audits change quarters. If your team is touching eight files for a one-field change, you do not have a performance problem, you have a coupling problem — same audit, different probe set: churn × complexity hotspots, dependency staleness, mutation score on the domain core, ghost-code archaeology, half-finished migrations.

How is this different from flutter analyze or a code-quality SaaS?

flutter analyze tells you what is syntactically wrong. Code-quality SaaS tells you what is statistically unusual. The audit tells you what is structurally wrong — which decision, which subtree, which dependency, on which device. A linter sees tokens. I read pipelines.

What if the problem is too big for a 3-day audit?

Day 2 is a hard checkpoint. If the diagnosis has not bound your symptom to a named engine-level cause by then, you get the partial trace and a full refund. The 3-day audit is sized to diagnose, not to rewrite. If your problem genuinely needs ten engineer-days, the Day-2 report says so with the scoped follow-on, and we both move forward with the right shape.

Do you do the fixes too?

The audit is the product. Your engineers typically ship the fix list faster than I would — the audit names the file and the line, and the team learns the codebase along the way. For specialist work (custom Impeller-compatible shader, vector_math_64 gesture rewrite, Impeller-Vulkan PSO warmup harness, large-scale state-management migration) I take implementation engagements after the audit, scoped per item.

Do you handle legacy Flutter codebases?

That is most of the work. Half-migrated null safety, v1 Android embedding holdouts, Material 2 surfaces under a Material 3 root, abandoned isolates, three state-management libraries in one repo, dependencies frozen since 2023. Legacy archaeology has its own probe set on Day 1 — the audit ends with a sequenced migration plan where every step is bound to a regression test so nothing silently breaks while the codebase moves.
